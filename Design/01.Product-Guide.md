# Dragon Alpha Product Guide

Last updated: 2026-03-22.

## Purpose
- Define what Dragon Alpha is, why it exists, and what this standalone migration should become.
- Keep top-level product direction, playable scope, and platform focus in one durable place.
- Give later design guides and implementation work a clear authority surface instead of letting chat history define the product.

## Product Summary
- Dragon Alpha is a standalone modernization of the original Dragon Alpha RPG, preserving the legacy game's world map, battles, progression, and recovered content where practical while moving onto a repo-owned Apple runtime.
- The player travels across area maps, triggers scripted interactions, shops, trains, fights battles, manages party state, and advances progression through a readable modern scene flow.
- The intended feel is a compact classic RPG with recognizable legacy content, dependable controls, and maintainable local ownership of code and resources.

## Product Vision
- Make Dragon Alpha playable again on current Apple hardware without depending on Carbon-era UI, resource-fork runtime assumptions, legacy package-based startup, or sibling-repo runtime code.
- Preserve the game's distinct legacy identity while improving layout safety, input reliability, and long-term maintainability.
- Prove a stable standalone macOS baseline first, then finish the public-release cleanup and any remaining platform retirement work.

## Player Promise
- Legacy Dragon Alpha content should feel present, not replaced by generic stand-ins.
- Core scenes should stay readable and usable on current Macs.
- Traversal, battle, status, and save/load should be dependable before broader polish or expansion work.

## Design Pillars
- Preserve legacy content and progression flow where practical.
- Own runtime code and resources locally inside Dragon Alpha.
- Ship incremental playable milestones instead of speculative large rewrites.

## Product Structure
- Core loop: launch into splash/front door, start or load a save, traverse the world map, resolve event interactions, enter battles, manage status/inventory, save progress, and continue exploration.
- Supporting structure: loose runtime resources, generated legacy-data headers, and focused validators protect parity and stability without redefining the player-facing game.
- The current build should first prove stable launch, navigation through the core scenes, and preservation of progression-critical flows on macOS.

## Platform And Release Direction
- Primary product target: `macOS`
- Deferred targets: `Android`, `Web`, `Windows`

## Current Scope
- Current implementation focus: keep Dragon Alpha compiling and running from local code/resources only, finish the remaining GitHub-signoff migration work, and preserve the playable baseline.
- Current design focus: document the standalone direction, convert repeatable checks into local deterministic validation where feasible, and keep the open-source path clear.
- Active content focus: splash/front door, new game and avatar flow, world-map traversal and interactions, battle flow, status flow, save/load, and loose legacy art/audio resources.

## Current Milestone
### First Playable Objective
- Launch Dragon Alpha from Xcode on macOS using only repo-owned runtime code and loose runtime resources.

### First Playable Acceptance
- Dragon Alpha builds successfully for macOS from local `Code/` engine/runtime files, with the owned engine surface now using local `Engine*` files and `E*` naming.
- The app launches into a stable splash/front-door flow and can reach New Game, world map, battle, status, and save/load without immediate runtime failure.
- The Xcode project, runtime entrypoints, and active validators no longer depend on sibling-repo engine/runtime checkouts or `Resources/Main.pkg`.

### First Playable Out Of Scope
- `Android`, `Web`, or `Windows` ports.
- Advanced music mixing/polish beyond the current scene-based runtime MIDI routing.
- Broad HD art refresh or non-legacy feature expansion.

## Next Product Step
- Finish the public-push signoff phase: treat `Design/Testing.md` as the primary visual-validation record, close the smaller native-only remainder from `Design/Release-Readiness.md` with screenshots and notes, get one capture-capable visual proof pass, and then move the stripped release tree into the real public git checkout while keeping any private migration helpers out of the public repo.

## Current Non-Goals
- Editing sibling repositories as part of Dragon Alpha migration unless the user explicitly asks for that.
- Recreating Carbon-era menus or bundle/resource-fork runtime behavior.
- Reintroducing legacy package-based runtime startup as a required dependency.

## Runtime Music Direction
- Long-term runtime music format: repo-owned Standard MIDI files (`.mid`).
- Long-term runtime path: `Resources/Music/`, with stable runtime filenames matching `Design/MidiRuntimeMapping.md`.
- Source-of-truth staging remains `Assets/Music/Original/MIDI/`; runtime copies should be produced locally from that staged MIDI set instead of depending on sibling repositories or legacy package payloads.
- Playback integration remains a later implementation step, but the format/path decision is now fixed so future music wiring targets one local contract.
